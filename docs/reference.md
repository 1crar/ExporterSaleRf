# Справочник по проекту

Этот документ описывает модули, функции, сущности и побочные эффекты AutoDoc Accounting.

## Структура файлов

| Файл | Роль |
| --- | --- |
| `__main__.py` | Запускает `main()` при `python -m autodoc_accounting`. |
| `__init__.py` | Экспортирует ленивую точку входа `main()`. |
| `app.py` | Главный сценарий обработки сделок. |
| `bitrix.py` | Адаптер Bitrix24 REST API. |
| `settings.py` | Общие константы: webhook Bitrix24 и корень серверной папки. |
| `parsing/utils.py` | Простые преобразования дат и сумм. |
| `parsing/documents.py` | Парсинг разрешительных документов и УПД. |
| `pdf/fields.py` | Модели текстовых полей PDF. |
| `pdf/recognize.py` | Извлечение и сортировка текстовых блоков PDF. |
| `pdf/extract.py` | Единая функция извлечения текста и таблиц из PDF. |
| `onec/documents.py` | Основная бизнес-логика работы с 1C. |
| `onec/printing.py` | Печать через внешнюю обработку `Print.epf`. |

## `app.py`

### `main() -> None`

Главный оркестратор.

Что делает:

- получает список сделок Bitrix24 через `findDeal()`;
- для каждой сделки забирает разрешение МИИР РК, CMR-данные и серверную папку;
- пропускает сделки без обязательных документов или с частичной отгрузкой;
- подключается к 1C через `V83.COMConnector`;
- парсит разрешение и УПД;
- **перед вызовом `operations1C()` переводит сделку в промежуточную стадию `C17:UC_BBPEXS`**;
- вызывает `operations1C()` для создания документов;
- обновляет Bitrix24: путь папки, файлы, стадия сделки;
- при успехе переводит сделку в `C17:UC_DHCK5O`.

Ключевые зависимости:

- `bitrix.findDeal()`, `getFile()`, `getCMRInfo()`, `changeState()`, `updateFolderBTX()`, `addToBTX()`;
- `parsing.documents.extractBitrixDocInfo()`;
- `parsing.documents.extractServerDocInfo()`;
- `onec.documents.operations1C()`;
- `win32com.client`.

Побочные эффекты:

- меняет стадии сделок Bitrix24 (промежуточная `C17:UC_BBPEXS` до обработки, финальная `C17:UC_DHCK5O` после);
- скачивает `./doc.pdf`;
- открывает COM-сессию 1C;
- создает документы в 1C;
- пишет файлы на сетевой диск;
- отправляет уведомления в чат.

## `bitrix.py`

### `send_msg_to_bot(msg)`

Отправляет сообщение в Bitrix24-чат через REST-метод `im.message.add`.

Параметры:

- `msg` - текст сообщения.

Возврат:

- ничего не возвращает.

Побочные эффекты:

- HTTP POST в Bitrix24.

### `findDeal()`

Ищет сделки Bitrix24 на стадии `C17:UC_NZTLLE`.

Возврат:

- список сделок из `response.json()["result"]`;
- пустой список, если сделок нет.

### `changeState(id, stageid)`

Обновляет стадию сделки.

Параметры:

- `id` - ID сделки Bitrix24;
- `stageid` - целевая стадия.

Побочные эффекты:

- вызывает `crm.deal.update`;
- печатает HTTP-ответ.

### `getFile(id, login, pw)`

Проверяет, требуется ли разрешительный документ МИИР РК, и при необходимости скачивает его.

Логика:

- читает поле `UF_CRM_1600310015`;
- ищет в значениях строку `миир рк`;
- если МИИР РК не найден, возвращает `"ok"`;
- если МИИР РК найден, пытается скачать первый файл из `UF_CRM_1651805106038`;
- если файл отсутствует, возвращает `"miir_no_doc"`;
- если файл скачан, сохраняет его в `./doc.pdf` и возвращает путь.

Возврат:

- `"ok"` - разрешение не требуется;
- `"miir_no_doc"` - разрешение требуется, но файл не вложен;
- `"./doc.pdf"` - файл скачан.

Формально функция не возвращает `None` явно (старый `return None` закомментирован), но `app.py` проверяет `docdealfile is None` — это страхует от случая, когда HTTP-запрос или разбор JSON внутри `getFile()` бросит необработанное исключение.

Побочные эффекты:

- HTTP-запросы к Bitrix24;
- запись `doc.pdf` в текущий каталог.

### `getCMRInfo(id)`

Читает из сделки данные для CMR и пути документов.

Возврат:

`targetServerPath, courierOrg, endCity, startCity, autoNum, RTUDate, isPartial`

Поля Bitrix24:

| Поле | Назначение по коду |
| --- | --- |
| `UF_CRM_1683719576` | Путь серверной папки сделки. |
| `UF_CRM_1575044812` | Перевозчик. |
| `UF_CRM_1575044651` | `startCity` в коде. |
| `UF_CRM_1575044638` | `endCity` в коде. |
| `UF_CRM_1681272900997` | Номер автомобиля/прицепа. |
| `UF_CRM_1626069863` | Дата РТУ. |
| `UF_CRM_1627529140` | Признак частичной отгрузки. |

Особенности:

- путь `S:\...` заменяется на UNC-корень из `SERVER_PATH`;
- **города в CMR используются инвертированно**: `endCity` записывается в `ГородПогрузки`, а `startCity` — в `ГородРазгрузки`. Это либо отражает бизнес-логику (город отправления — погрузка из точки зрения перевозчика), либо является ошибкой в коде.

### `addToBTX(upd, cmr, dealID)`

Добавляет готовые PDF к сделке Bitrix24.

Параметры:

- `upd` - путь к PDF УПД/РТУ;
- `cmr` - путь к PDF CMR;
- `dealID` - ID сделки.

Логика:

- читает существующие файлы из `UF_CRM_1593667879`;
- кодирует новые PDF в base64;
- вызывает `crm.item.update`.

### `updateFolderBTX(dealID, folderPath)`

Записывает путь клиентской папки в поле `UF_CRM_1701228716877`.

## `parsing/utils.py`

### `convDate(date)`

Преобразует дату из текстового формата УПД в `dd.mm.yyyy`.

Поддерживает:

- даты через точку или запятую;
- русские сокращения месяцев: `янв`, `фев`, `мар`, `апр`, `июн`, `июл`, `авг`, `сен`, `окт`, `ноя`, `дек`;
- месяц `май` явно не обрабатывается, но корректно попадает в `else`-ветку, которая по умолчанию трактует неизвестный месяц как `.05.`.

### `convDateTo1CFormat(date)`

Преобразует строку `dd.mm.yyyy` в `datetime` с UTC timezone.

Используется перед записью дат в 1C.

### `convSum(docSum)`

Нормализует сумму:

- убирает пробелы;
- заменяет десятичную запятую на точку;
- возвращает строку с числом.

## `parsing/documents.py`

### Глобальная переменная `dealID`

Legacy-глобальная переменная для сообщений об ошибках. Перед парсингом `app.main()` записывает в нее текущий ID сделки.

### `extractBitrixDocInfo(myFile)`

Извлекает данные из разрешительного PDF, скачанного из Bitrix24.

Параметры:

- `myFile` - путь к PDF.

Логика:

- читает текст первой страницы через `PyPDF2`;
- определяет организацию по маркерам в тексте;
- извлекает таблицы через `camelot`;
- поддерживает два типа документа: заключение об идентификации и альтернативный формат;
- собирает наименования товаров и коды ТН ВЭД;
- чистит переносы строк, пробелы и пустые значения.

Возврат:

`orgName, items, itemsCodes`

### `extractServerDocInfo(serverPath)`

Ищет и разбирает УПД в папке сделки.

Параметры:

- `serverPath` - папка сделки на серверном диске.

Логика:

- ищет PDF с `упд` или `универсальный` в имени;
- если по имени не найдено, проверяет все PDF и ищет текст `передаточный`;
- извлекает текст и таблицы через `pdf.extract.get_info()`;
- при необходимости повторяет извлечение с поворотом страницы;
- регулярными выражениями ищет номер, дату и суммы УПД;
- через `camelot` и `pdfplumber` собирает табличную часть;
- формирует массив ГТД;
- сохраняет диагностические таблицы в Excel.

Возврат:

`updNum, updDate, updSumItems, updSumNDS, updSum, gtdArr, df`

Если УПД не найден или не читается, возвращает семь `None`.

Побочные эффекты:

- может создавать и удалять `rotated.pdf`;
- пишет `test.xlsx` и `test2.xlsx`;
- отправляет уведомления в чат при ошибках.

## `onec/documents.py`

Этот файл содержит основную бизнес-логику 1C. Почти все функции работают с COM-объектами 1C и имеют сильные побочные эффекты.

### `correctord(objSup, dfUpd, dfSum, connection)`

Корректирует заказ поставщику по табличной части УПД.

Параметры:

- `objSup` - ссылка или объект заказа поставщику;
- `dfUpd` - таблица УПД;
- `dfSum` - сумма УПД;
- `connection` - COM-подключение к 1C.

Логика:

- приводит числовые колонки УПД к формату с точкой;
- считает сумму строк;
- сопоставляет товары УПД с номенклатурой заказа поставщику;
- применяет fuzzy-сравнение через `thefuzz.ratio`;
- специальные услуги мапит на `Упаковка` или `Доставка (ТЭУ)`;
- если сумма совпала и все товары найдены, очищает табличную часть заказа и заполняет ее по УПД;
- выставляет ставки НДС по тексту из УПД (поддерживаются 22%, 20%, 18% и 0%);
- записывает заказ.

Побочные эффекты:

- изменяет заказ поставщику в 1C;
- может создать `testik.xlsx`.

### `createPTU(connection, dosSs, dealID, RTUdate)`

Создает документ `ПриобретениеТоваровУслуг` для цепочки перепродажи.

Параметры:

- `connection` - COM-подключение;
- `dosSs` - заказ клиента;
- `dealID` - ID сделки;
- `RTUdate` - дата РТУ.

Логика:

- строит запрос 1C по КП, заказу клиента и уже проведенным реализациям;
- рассчитывает остатки к оформлению;
- заполняет организацию, склад, договор, контрагента, валюту, НДС и табличную часть;
- проводит документ.

Возврат:

- ссылка на созданный ПТУ.

Нюанс реализации:

- в сигнатуре аргумент называется `RTUdate`, но внутри функции используется `RTUDate`. Фактически `NameError` не возникает, потому что `createPTU()` вызывается только из `operations1C()`, и Python подхватывает `RTUDate` из объемлющего scope через замыкание. Однако аргумент `RTUdate` остается неиспользованным, и при вызове `createPTU()` вне `operations1C()` произошел бы `NameError`.

### `createRTU(connection, dosSs, dealID, RTUDate)`

Создает документ `РеализацияТоваровУслуг` для перепродажи.

Логика:

- использует похожий запрос к КП, заказу клиента и реализациям;
- заполняет реализацию на экспорт;
- выставляет НДС 0%;
- проставляет склад, цены, суммы и коды ТН ВЭД;
- записывает и проводит документ.

Возврат:

- объект созданной РТУ.

### `operations1C(dealID, updSum, updNum, updDate, connection, courierOrg, endCity, startCity, gtdArr, dfUpd, RTUdate, autoNum, has_miir_doc=True)`

Главная функция интеграции с 1C.

Параметры:

- `dealID` - номер сделки Bitrix24 и номер заказа клиента в 1C;
- `updSum`, `updNum`, `updDate` - сумма, номер и дата УПД;
- `connection` - COM-подключение к 1C;
- `courierOrg` - перевозчик для CMR;
- `endCity`, `startCity` - города для CMR;
- `gtdArr` - массив ГТД из УПД;
- `dfUpd` - табличная часть УПД;
- `RTUdate` - дата РТУ из Bitrix24;
- `autoNum` - номер авто/прицепа;
- `has_miir_doc` - признак наличия разрешительного документа МИИР РК.

Основная логика:

- ищет `ЗаказКлиента` по номеру сделки;
- определяет, есть ли внутренняя перепродажа;
- проверяет наличие разрешения МИИР РК для перепродажи;
- ищет `ЗаказПоставщику` по `ИДСделкиБитрикс24`;
- ищет `КоммерческоеПредложениеКлиенту`;
- при отсутствии заказа поставщику создает его из заказа клиента и КП;
- если сумма УПД отличается от суммы заказа поставщику, перестраивает табличную часть заказа;
- вызывает `correctord()` для финальной коррекции по УПД;
- создает и проводит ПТУ поставщика;
- создает входящий счет-фактуру через `createInvoice()`;
- при перепродаже создает дополнительный ПТУ через `createPTU()`, РТУ через `createRTU()`, исходящий счет-фактуру `СчетФактураВыданный` и CMR; клиентская папка определяется по жестко зашитой маппинг-таблице организаций;
- без перепродажи обновляет `ВариантОбеспечения` товаров заказа клиента на `Отгрузить`, создает РТУ напрямую по заказу клиента, **создает `СчетФактураВыданный`** и CMR; клиентская папка определяется поиском по файловой системе через `Path.rglob(dealID)`;
- печатает документы через `print_docs()`.

Возврат:

- путь клиентской папки с напечатанными PDF;
- `None`, если обработка не может быть завершена.

Побочные эффекты:

- создает и изменяет документы 1C;
- создает элементы справочника `НомераГТД`, если номер не найден;
- создает папки на сетевом диске;
- печатает PDF через внешнюю обработку;
- отправляет сообщения в Bitrix24-чат.

### `createInvoice(docSs, updDate, updNum, connection)`

Создает входящий счет-фактуру `СчетФактураПолученный`.

Параметры:

- `docSs` - документ-основание;
- `updDate` - дата УПД;
- `updNum` - номер УПД;
- `connection` - COM-подключение.

Логика:

- заполняет организацию, контрагента, валюту, ИНН/КПП;
- добавляет документ-основание;
- рассчитывает сумму НДС;
- проводит счет-фактуру.

### `check1Cdocdeal(dealID, orgname, itemcodes, connection)`

Сверяет заказ клиента 1C с разрешительным документом из Bitrix24.

Логика:

- ищет заказ клиента по номеру сделки;
- сравнивает организацию с `orgname`;
- сравнивает коды ТН ВЭД строк заказа с `itemcodes`.

Возврат:

- `True`, если данные совпали;
- `False`, если заказ не найден или есть расхождения.

Сейчас вызов в `app.py` закомментирован.

### `testupdprint()`

Ручная тестовая функция для печати РТУ и CMR через `Print.epf`.

Использует зашитое подключение к тестовой базе 1C и конкретные номера документов.

## `onec/printing.py`

### `print_docs(cmrDoc, RTUdoc, foundServerPath, connection)`

Печатает CMR и УПД/РТУ через внешнюю обработку 1C.

Параметры:

- `cmrDoc` - документ CMR;
- `RTUdoc` - документ реализации или счет-фактура, используемый печатной формой;
- `foundServerPath` - папка вывода без имени файла;
- `connection` - COM-подключение.

Логика:

- создает внешнюю обработку `Print.epf`;
- вызывает `Print()` для CMR дважды: с аргументом `1` и `2` (предположительно оригинал и копия);
- вызывает `Print4()` для УПД/РТУ дважды: с аргументом `1` и `2`;
- ожидаемый результат — файлы `cmr.pdf` и `rtu_upd.pdf` в папке `foundServerPath`.

В файле также оставлен большой закомментированный фрагмент старой логики ручной печати CMR.

## Пакетные `__init__.py`

### Корневой `__init__.py`

Экспортирует `main()` и лениво импортирует `app.main`.

### `parsing/__init__.py`

Экспортирует утилиты сразу, а тяжелые функции парсинга документов грузит лениво через `__getattr__`.

Причина:

- не загружать PDF-зависимости при простом импорте `parsing`.

### `onec/__init__.py`

Экспортирует функции 1C и печати лениво через `__getattr__`.

Причина:

- не загружать `win32com`, `pandas` и остальную тяжелую логику при простом импорте пакета.

### `pdf/__init__.py`

Объявляет доступные подмодули. Тяжелые зависимости импортируются напрямую вызывающим кодом.

## Ошибки и ручная обработка

Сделка обычно уходит в ручную обработку, если:

- нет обязательного разрешительного документа;
- указана частичная отгрузка;
- УПД не найден, является сканом или имеет неподдерживаемый формат;
- не найден заказ клиента в 1C;
- не совпадают организации КП и заказа клиента при перепродаже;
- МИИР РК без документа разрешения, но сделка с перепродажей;
- не удается провести документы в 1C;
- невозможно определить клиентскую папку или сформировать PDF.

Для ручной обработки используется стадия `C17:UC_BBPEXS`.

Важно: в текущем коде `app.main()` переводит сделку в `C17:UC_BBPEXS` **до** вызова `operations1C()`. Это значит, что каждая обрабатываемая сделка сначала попадает в «ручную» стадию, и только при успешном завершении переводится в `C17:UC_DHCK5O`.

## Замеченные несостыковки

Эти пункты не исправлены в рамках документации, но важны для следующего технического долга:

- `app.main()` не передает `has_miir_doc=False` в `operations1C()` для результата `getFile() == "miir_no_doc"`, хотя внутри `operations1C()` уже есть логика запрета перепродажи без разрешения.
- `createPTU()` принимает `RTUdate`, но использует `RTUDate`. Ошибка не проявляется благодаря замыканию (вызов только из `operations1C()`), но аргумент функции фактически игнорируется.
- `getCMRInfo()` возвращает `startCity` и `endCity`, но в `operations1C()` они записываются в CMR инвертированно: `endCity` → `ГородПогрузки`, `startCity` → `ГородРазгрузки`. Требуется проверить, корректно ли это с точки зрения бизнес-логики.
- Секреты Bitrix24 и учетные данные 1C хранятся в коде.
- Диагностические Excel/PDF-файлы пишутся в рабочую директорию без уникальных имен, поэтому параллельный запуск может перетирать файлы.
- Аннотация типа `sortFields()` указывает `list[list[MF]]`, но функция возвращает плоский `list[MF]`.
